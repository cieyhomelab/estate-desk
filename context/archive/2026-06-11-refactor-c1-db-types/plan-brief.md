# Refactor C1: Generated DB Types — Plan Brief

> Full plan: `context/changes/refactor-c1-db-types/plan.md`
> Research: `context/changes/refactor-opportunities/research.md`

## What & Why

The Supabase client has no `<Database>` generic, so every query returns `any`-shaped rows.
Domain types `Listing` and `TransactionSnapshot` are hand-mirrored in two files, with
`commission_percent` typed in 4 separate places. This is the amplifier under the other debts:
every stringly-typed `.select()` / `.update()` / `.insert()` call has no schema backing,
so column renames and nullability changes propagate manually. The Supabase CLI is already
wired — generation was simply never set up.

## Starting Point

`src/lib/supabase.ts:9` constructs the client untyped. `src/types/listings.ts` and
`src/types/transaction.ts` hold hand-written mirrors of the DB schema. There are 34 `.select(`,
8 `.update(`, and 7 `.insert(` call sites returning `any`. No `database.types.ts` exists.
`astro check` (CI gate at `ci.yml:21`) is the safety net for all three phases.

## Desired End State

`src/types/database.types.ts` is the single source of truth, generated by `npm run db:gen-types`.
`Listing` and `TransactionSnapshot` are aliases of `Tables<'listings'>` and
`Tables<'transaction_snapshots'>`. The client is typed `<Database>`. `astro check` passes clean.

## Key Decisions Made

| Decision | Choice | Why (1 sentence) | Source |
|---|---|---|---|
| Phasing | Generate → alias → type client (3 phases) | Each phase is independently verifiable; no phase forces a big-bang migration | Plan |
| Alias vs rewrite | Alias (preserve export names) | 11 consumers don't need to change; the alias is a type-level operation | Plan |
| Call-site migration | Opportunistic in Phase 3 | Not all 49 call sites need touching now; astro check surfaces the ones that matter | Plan |
| Validation layer | Not included | C5 is a separate plan | Research |

## Scope

**In scope:**
- `db:gen-types` npm script
- `src/types/database.types.ts` (generated, committed)
- `Listing` and `TransactionSnapshot` aliased onto generated types
- Inline `.single<{commission_percent…}>()` cast removal in `close.ts:35`
- `<Database>` on the Supabase client factory
- Call-site fixes surfaced by `astro check`

**Out of scope:**
- Forced migration of all 49 call sites
- Validation layer / schema (C5)
- DB migrations or RLS policy changes

## Architecture / Approach

Three additive phases, each gated by `astro check`. Phase 1 generates and commits the type
file (zero code impact). Phase 2 aliases the domain types — `astro check` surfaces any
hand-written drift as real errors to be resolved. Phase 3 types the client factory (one line),
then opportunistically fixes the type errors that propagate through call sites.

## Phases at a Glance

| Phase | What it delivers | Key risk |
|---|---|---|
| 1. Generate + Commit | database.types.ts on disk; generation script | Schema drift surfaces in Phase 2 (expected, not a risk) |
| 2. Alias Domain Types | Listing / TransactionSnapshot backed by generated types | Hand-written types may differ from DB reality — astro check surfaces this |
| 3. Type Client + Migrate Sites | Client typed <Database>; inferred types at call sites | astro check may surface unexpected type errors requiring non-trivial fixes |

**Prerequisites:** Local Supabase running for `supabase gen types --local`.
**Estimated effort:** ~1-2 sessions (Phase 1 is trivial; Phase 3 length depends on call-site errors).

## Open Risks & Assumptions

- The generated `Listing` type may differ from the hand-written one (e.g., nullability); `astro check` will surface the differences as errors to be resolved, not suppressed.
- `commission_percent` is currently `number | null` in the DB; the hand-written types may have it as `number` — needs resolution in Phase 2.
- Phase 3 call-site fixes may be non-trivial for dynamic `.select()` strings; those can be deferred with a scoped comment.

## Success Criteria (Summary)

- `npm run db:gen-types` regenerates the file cleanly.
- `Listing.commission_percent` inferred from the DB schema, not from a hand-written type.
- In an IDE, `.from('listings').select()` returns a typed row, not `any`.
